# Changelog

All notable changes to **torrent-search-go** (Stremio protocol API, scrapers, jobs) are documented here.

Format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).
Stremio manifest versions track the addon release line (`internal/stremio/manifest.go` `addonVersion`, overridable by `ADDON_VERSION`). The source of truth for the shipped addon version is `tpb-stremio-addon/package.json`.

## [Unreleased]

### Security
- **Auth bypass via unsigned "temporary token" removed (Critical)** - `internal/middleware/auth.go` accepted any base64-encoded `{id,email,name,timestamp}` JSON blob as a valid session with no signature check, letting an attacker forge any user identity and take over their account (favorites, Real-Debrid key, all `RequireAuth` routes). The `validateTemporaryToken` fallback and its base64/email forging path have been deleted; only database-issued sessions validate now.
- **Monitoring endpoints fail-closed in production (Critical)** - `/api/monitoring/*` (IP block/unblock, cache bust, log viewer, job triggers, debug-favorites) shipped with zero auth when `DASHBOARD_PASSWORD` and `MONITORING_IP_ALLOWLIST` were unset, the documented default. `RequireDashboardAuth` (`internal/middleware/dashboard_auth.go`) now denies all monitoring traffic in production when no password is configured; dev/local stays open.
- **SSRF defense on all outbound scraper traffic (Critical)** - a user-supplied `torrentUrl` path segment reached scrapers (`nyaa`, `hiddenbay`, `pornrips`, ...) as a raw request URL, enabling server-side requests to `169.254.169.254`, internal services, and localhost. New `internal/services/scraper/safehttp.go` provides `NewSafeClient`: a dial-time IP filter (refuses private/loopback/link-local/CGNAT/metadata ranges, pins the resolved IP to defeat DNS rebinding), a `CheckRedirect` that re-validates each hop, and an 8 MiB response-body cap. Every scraper constructor and the shared `Service` client now route through it.
- **Stremio `:config` segment redacted from logs and the API tracker (High)** - `/stremio/:config/...` and `/magnetio/:config/...` URLs carry the user's Real-Debrid / TPDB / StashDB / media-flow keys in the path; every catalog/meta hit wrote them into `app.log`, `all.log`, and the `api_usage` Mongo collection. `redactPath` (`internal/middleware/router.go`) replaces the config segment with `:redacted` before logging or tracking.
- **`X-Forwarded-For` / `X-Forwarded-Proto` gated on `TrustProxy` (High)** - `clientIP`, `isHTTPS`, and the IP-allowlist resolver honored proxy headers unconditionally, so a direct client could spoof its IP past the rate limiter, DDoS guard, and monitoring allowlist, or force HSTS over plain HTTP. A package-level `trustProxyHeaders` flag (set from `cfg.Security.TrustProxy` via `WithTrustProxy`, now actually wired instead of the prior no-op stub) gates all three.
- **CORS no longer reflects arbitrary origins with credentials (High)** - when `FRONTEND_URL`/`ADDITIONAL_CORS_ORIGINS` were unset, config fell back to `["*"]` and the middleware reflected the requesting origin alongside `Access-Control-Allow-Credentials: true`, letting any website issue authenticated cross-origin requests with the victim's session cookie. `isOriginAllowed` (`internal/middleware/cors.go`) now ignores `*` when credentials are enabled and requires an explicit origin match.
- **Global request-body cap (High)** - ~22 JSON endpoints had no `MaxBytesReader`, so a single multi-GB body could exhaust memory. A new `MaxBodySize` middleware + `WithMaxBodySize` router option caps every request at 2 MiB (`main.go`); `KVSet` additionally bounds a single cache value at 256 KiB so one user can't bloat the shared KV collection.
- **IDOR on favorite entry mutations closed (Medium)** - `UpdateFavoriteEntryCoverImage` and `UpdateFavoriteEntryMagnetLink` filtered Mongo by `_id` only, so any authenticated user could overwrite another user's cover URL or swap their magnet to a malicious torrent by guessing the entry id. Both storage methods now take a `userID` and filter `{_id, user_id}`; the handlers pass the authenticated user's id.
- **Constant-time `ADDON_API_TOKEN` comparison (Medium)** - `MatchesAddonToken` compared the service token with `==`, exposing a timing oracle. Switched to `subtle.ConstantTimeCompare`.

### Changed
- **Concurrency: `JobScheduler.taskLocks` map guarded against data race** - the lazy-init `taskLock` was called from concurrent job goroutines with no synchronization (`fatal error: concurrent map read and map write`). A `taskLocksMu sync.Mutex` (`internal/handlers/monitoring.go`) now guards the map.
- **Concurrency: Mongo `Client.isConnected` / `lastCheck` made atomic** - written by `HealthCheck` from background jobs and read by `GetStats` from concurrent HTTP handlers, the two fields were a `-race` data race. Switched to `atomic.Bool` / `atomic.Pointer[time.Time]` (`pkg/mongo/client.go`).
- **DDoS guard `traffic` map gets a periodic sweep** - every distinct client IP got a permanent map entry with no eviction, an unbounded memory leak on a public API. `sweepTraffic` (`internal/middleware/ddos.go`) now runs each sync tick and deletes entries whose pruned timestamps are empty.
- **Addon manifest description refreshed; addonVersion baseline bumped to 1.9.75** - manifestDescription now leads with sources, debrid providers (14, top three named), MediaFlow proxy, and encrypted cross-device presets instead of platform-first phrasing, and the addonVersion const (`internal/stremio/manifest.go`) is realigned to the current edge release (`tpb-stremio-addon` package.json 1.9.75). Still env-overridable via `ADDON_VERSION` and stamped by the edge at runtime, so only direct-backend manifest hits are affected.
- **AllDebrid and PikPak removed from the TPB debrid provider set (14 → 12); AllDebrid removed from the Magnetio moch set (8 → 7)** - AllDebrid rejects magnet uploads from VPS/datacenter egress (`NO_SERVER` block on the prod Hetzner IP), so `adKey` installs never cached and only P2P showed; PikPak is dropped alongside for a consistent TPB provider set. `internal/stremio/debrid.go` drops the `adKey`/`pkKey` `DebridProviders` and `DebridKeyFields` entries plus the `debridKey`/`setDebridKey` switch cases; `internal/stremio/config.go` drops the `AdKey`/`PkKey` Config fields (existing install URLs carrying them still parse, the fields are just ignored). `internal/stremio/manifest.go` description now reads "12 debrid providers (Real-Debrid, TorBox, and more)". `pkg/mongo/addon_status.go` feature body drops AllDebrid and reads "9 more providers". The Magnetio proxy (`internal/stremio/magnetio/{config,manifest}.go`) drops the `ad` `MochOption`, the `AllDebridApiKey` Config field + its parse/lookup cases, and the `ad` idPrefix from the catalog/meta resources. `addonVersion` const realigned to 1.9.77 to match the edge (`tpb-stremio-addon` package.json 1.9.77). `go build ./...` clean; `go test ./internal/stremio/...` passes.

### Added
- **Enriched-scenes Mongo store backs the TPDB/StashDB catalogs and the `porndb:`/`stash:` meta+stream path** - a new `enriched_scenes` collection (`pkg/mongo/enriched_scenes.go`, `pkg/models/enriched_scene.go`) holds every TPDB/StashDB scene the background job matched to a resolvable torrent from the user's configured sources, keyed `_id` = the scene's Stremio metaID. `serveTPDBCatalog` (tpdb_new browse), `serveTPDBMeta`/`tpdbStreams`, and `serveCategoryCatalog` (tpdb_cat/stashdb_cat, `internal/stremio/catalog.go`) read from the store instead of hitting live TPDB/StashDB on every cache miss; `tpdb_search` stays live and fires a bounded on-demand upsert (`EnrichedScenesOnDemand`, `internal/stremio/handler.go`) so a first search populates the store and subsequent browses are store-backed. Per-source best torrents are stored on the doc (`torrents.<src>`), so the stream path emits without a second query. Metadata fields are `$setOnInsert` (immutable; TPDB/Stash scene data is immutable), `matched_sources`/`attempted_sources` are `$addToSet $each`, per-source torrents `$set` by dotted key. Cold store (no `EnrichedScenes` wired) returns empty for browse; meta/stream keep a live `GetScene` fallback until the store is confirmed complete. The pre-populated store was bulk-filled once from the Mac against prod Mongo (~59.5k scenes, 112 ticks); the deployed job runs a shallow recent sweep.
- **`EnrichedScenesSync` background job + on-demand populator** - new `internal/services/jobs/enriched_scenes_sync.go` mirrors `pornrips_enrich.go` (comma-separated TPDB/StashDB key pool, worker pool, transient-failure skip). Per tick it discovers TPDB BrowseScenes + StashDB per-category scenes and fans the configured source-set (`ENRICHED_SCENES_SOURCES`) at each scene via `findMatchingTorrent` + `metadata.VerifyMatch`, upserting the per-source best torrent. Wired into the scheduler (`internal/handlers/monitoring.go` `startEnrichedScenesSync`/`TriggerEnrichedScenesSync`, `internal/config/config.go` `EnrichedScenesSync` job, `ENRICHED_SCENES_SYNC_INTERVAL_MS` default 10min / `_INITIAL_MS` 2min) with a manual trigger route `POST /api/monitoring/enriched-scenes-sync-trigger` (`main.go`). Local bulk-fill tooling (`cmd/tpbingest`, `cmd/tpbcount`, `cmd/tpbrematch`) pre-seeds historical data; `cmd/` is gitignored.
- **`STASHDB_MIN_GAP_MS` env override** - `internal/services/metadata/stashdb.go` reads the throttle gap in the constructor mirroring `tpdb.go`, so a single StashDB key can exceed 4 req/s without adding keys.

### Fixed
- **Cover matching for dated TPB 4K entries no longer accepts wrong-studio/wrong-date scenes** - `VerifyMatch` (`internal/services/metadata/release_match.go`) gates every TPDB and StashDB candidate the cover/description cache job (`jobs/description_image_cache.go`) resolves. Its `parsed.Date != ""` branch ended in `return IsGoodMatch(candidate.Title, parsed.Scene)`, which ignored the date entirely: because `PerformerOverlap` falls back to the candidate *title* when a scene has no performers, a candidate from a different studio with a release date years off was accepted on a couple of overlapping scene words alone, slapping an unrelated provider poster onto a homepage 4K entry whose title already carried the correct `Studio YY MM DD` prefix. The branch now treats the release date as the hard discriminator: a candidate that carries its own date must be within 3 days (`dateClose`) or it is rejected regardless of title-token overlap; a dateless candidate (e.g. a movies record) must additionally agree on studio before title overlap is trusted. The legit network-naming case is preserved - a date-aligned scene with a good title still matches even when TPDB/StashDB names the parent network ("Blacked" -> "Vixen Media Group") rather than the torrent's studio token. Net effect: when studio and date do not corroborate, the bad provider cover is rejected and the entry falls back to its description-derived image automatically. Tests: `TestVerifyMatchRejectsWrongStudioWrongDate`, `TestVerifyMatchDatelessNeedsStudio`, `TestVerifyMatchStudioNameMismatchButDateLinesUp` (`release_match_studio_date_test.go`).
- **Search catalog no longer duplicated across multi-part installs** - `BuildManifest` appended the dedicated `search` catalog whenever `piratebay` was a source, so every part of a split install carried its own Search row (each part has `piratebay`). Now gated to `cfg.Group <= 1` (unsplit or part 1): Search resolves across all piratebay content regardless of chunk, so one instance suffices and the board no longer shows N duplicated Search rows that also broke the global sort across parts. Paired with the Node-edge `installBuilder` fix (TPDB/StashDB dedup via explicit empty category arrays on parts 2..N + alphabetical chunking) so a multi-part install, with parts installed in order, yields a globally sorted board: non-TPB-Studio catalogs first (alphabetical, part 1 only), then TPB-Studio catalogs (alphabetical across all parts). The TPDB/StashDB suppression relies on the existing `len(TpdbCategories) > 0` emission gate, now locked by `TestBuildManifestTpdbSuppressedByEmptyCategories` (explicit empty `TpdbCategories` + active server key suppresses the TPDB catalog; the edge sends `[]` on parts 2..N because omitting the field would let the backend fill default categories from the shared tpdb/stashdb key and re-emit on every part). Tests: `TestBuildManifestSearchOnlyOnPart1`, `TestBuildManifestTpdbSuppressedByEmptyCategories`.
- **`EnrichedScenesSync` StashDB discovery only walked one key, stopped after ~18 scenes, and dropped the walk on the first 429** - `discoverStashScenes` (`internal/services/jobs/enriched_scenes_sync.go`) had three prod bugs that left the StashDB half of `enriched_scenes` thin and skewed to the first category. (1) It built one `metadata.NewStashDBClient` from `stashKeys[0]`, so the comma-separated key pool never raised StashDB throughput - extra keys were dead weight. (2) The `discovered >= stashCats` cap counted *scenes*, not *categories*; the default `enrichedDefaultStashCats=18` stopped the walk after ~18 scenes (<1 category), so the deployed job re-discovered only a slice of the first category each tick instead of the full Default:true set. (3) `stash.FetchScenes` had no retry on transient 429/5xx, so a single rate-limit response dropped the rest of the categories. Rewritten to mirror `discoverTPDBScenes`: each key gets its own client (per-key throttle), `OrderedCategories()[:stashCats]` is the CATEGORIES walked (cap now bounds coverage, not scene count - bulk-fill's `STASH_CATS_PER_TICK=2000` exceeds the ~40-category set and covers all), the categories stride across the key pool in parallel (key i walks cats i, i+n, ...), and `FetchScenes` retries up to `enrichedDiscoverRetries=6` with `2*(try+1)s` backoff. With a single key the walk is identical to the old sequential one. StashDB `stashdb_cat` coverage now populates across all configured categories per tick instead of starving them.
- **Store-backed tpdb_cat/stashdb_cat genre catalogs returned zero for ~17 of 40 genres (regression from the store flip)** - `serveCategoryCatalog` (`internal/stremio/catalog.go`) filtered `tags_norm = NormToken(StashTag)`, but TPDB/StashDB tag scenes with compound forms the bare category name does not collapse to: "MILF" -> `milf30` (TPDB "MILF (30+)"), "Anal" -> `analsex` (StashDB "Anal Sex"), "Blonde" -> `blondhair`, "Feet" -> `footfetish`, "Tattoo" -> `tattoos`, "Teen" -> `teen1822`, "Latina" -> `latinwoman`, etc. So the MILF/Teen/Mature/Latina/Ebony/Blonde/Redhead/Petite/Tattoo/Double Penetration/Public/Stepfamily/Cuckold/Pissing/Feet genres (and StashDB Anal/POV) rendered empty in the addon discover section. The old `CategoryWarmer`-backed `tpdb_cat` used `TPDBQuery` as a TPB search term and never had this mismatch; the store flip introduced it. Fixed by a per-genre alias map `enrichedTagAliases` + `enrichedTagTokens(genre)` (curated from an `enriched_scenes` vocabulary audit, distinct from the pornrips `prTagAliases` since the two stores' tag vocabularies differ - e.g. enriched has `footfetish`=3381 vs pornrips `cumonfeet`); `serveCategoryCatalog` now queries `tags_norm $in [bare] + aliases`. `GetEnrichedScenesByMatchedSources` (`pkg/mongo/enriched_scenes.go`) takes `tags []string` instead of `tag string` (the `tpdb_new` browse path passes `nil` = no tag filter). Verified against prod: every previously-empty genre now returns healthy counts (MILF 5208/77, Teen 3650/56, Mature 951/24, Blonde 7871/110, Tattoo 5325/148, Feet 3764/64, Stepfamily 942/42, Cuckold 813/13, DP 1427/43, Anal-stash 119, etc.). A follow-up audit found two more mismatches in the same family: **Rough Sex** matched 1/0 because both stores tag with bare `rough` (TPDB 2786, StashDB 23), not `roughsex` - added alias `roughsex` -> `rough` (now 2787/23); **POV** matched 5086/0 because StashDB has no bare `pov`, only position-pov compounds (`malepov`, `cowgirlpov`, `doggystylepov`, ...) - added a `pov` alias set (now 5176/85). Petite stays 10/0 (genuine sparseness: StashDB has no petite tag at all, TPDB sparse even with aliases - real data availability, not a mismatch). `tpdb_new` (browse) and the "All" genre are unaffected (no tag filter).
- **`pr_tag` genre aliases expanded for the 5 pornrips genres that returned <10 streamable entries** - a full audit of every catalog sub-category (`cmd/tpbaudit`, gitignored, runs the SAME `tags_norm $in`/`studio_norm` + `info_hash` filters the deployed catalog code runs) flagged 5 `pr_tag` genres whose bare `NormToken` is dead in the pornrips vocabulary: **Stepfamily** (0 -> 17891), **Pissing** (0 -> 2050), **Rough Sex** (6 -> 14853), **Redhead** (2 -> 19936, the prior `naturalredheads` alias was itself dead), and **Ebony** (2 -> 1552). `prTagAliases` (`internal/stremio/pornrips_catalog.go`) now maps each to the compound `tags_norm` tokens the enrich sweep actually writes (e.g. `stepfamily` -> `stepmother`/`stepdaughter`/`stepsister`/...; `pissing` -> `pissinmouth`/`pissdrinking`/...; `roughsex` -> `rough`; `redhead` -> `redhairfemale`/`redhair`; `ebony` -> `blackwoman`/`ebonyonebonysex`/`blackonblack`). Curated from a `pornrips_entries` `tags_norm` vocabulary audit to exclude false positives (`black` is ambiguous hair/stockings/eyes, `blackmail` is crime fantasy, `coloredhair`/`redlipstick`/`stockingsred` are not redhead, `blackhair` is hair not skin) - distinct from the `enriched_scenes` `enrichedTagAliases` since the two stores' tag vocabularies differ. `prTagTokens` returns the aliases ONLY for aliased genres (the bare token is dead, so including it would add noise without matches); bare token for non-aliased genres is unchanged. Exported wrappers `EnrichedTagTokens`/`PornripsTagTokens`/`PornripsStudioOptions` added for the read-only audit tool so it replicates the deployed query without duplicating the alias/studio maps. Test: `TestPrTagTokens`.

### Removed
- **PornRips catalog/search reads no longer surface entries without a resolved torrent** - `findPornrips` (`pkg/mongo/pornrips_entries.go`, the shared read path behind `GetPornripsRecent`/`GetPornripsByStudio`/`GetPornripsByTag`/`SearchPornrips`/`GetPornripsEntriesByPerformer`) now requires a non-empty `info_hash` via the new pure helper `streamablePornripsFilter`. The torrent-backfill sweep writes `info_hash`+`torrent_url` together (`SetPornripsTorrent`), so an empty `info_hash` means the entry is not yet backfilled - the newest posts (~0.04% of the collection, all <2 days old) that the newest-first sweep is about to resolve. Hiding them keeps the pr_recent/pr_studio/pr_tag feeds and PornRips search to items a stream can actually be opened for; previously they surfaced as cards that, on click, could only resolve via a live Cloudflare-blocked detail-page fetch. The filter copies (does not mutate) the caller's filter map and is idempotent when the caller already sets `info_hash` (`GetPornripsEntriesByPerformers`). The sweep reads (`GetPornripsEntriesMissingEnrichment`/`MissingTorrent`) and `GetPornripsEntryBySlug` use their own `Find` and bypass this, so backfill and by-slug meta rendering still work for unswept entries. No new index needed - the existing `studio_norm`/`tags_norm`/`date:-1` index walks reject the ~134 unswept docs in-memory. Test: `TestStreamablePornripsFilter`.

### Fixed
- **torrent-search-ui showed wrong/missing TPB covers that the Stremio catalog resolved correctly** - the MetaEnricher (the background job that durably writes TPDB/Stash scene covers to `shared_meta` for the UI) used `SearchMetadata`, a single raw-title probe, while the Stremio catalog live path looped the full variant set (`metadataTitlesForLookup` + `SearchMetadataProbe`). For messy/JAV TPB titles the single probe mismatched or missed, so a wrong/empty poster was written to `shared_meta`, and the UI (no live probe to override it on read) showed it while the catalog looked correct (its live probe wrote the right poster to Redis first, so the enricher's poster-aware dedup then skipped the entry). The variant generator (`MetadataTitlesForLookup`) and the multi-probe loop (`SearchMetadataVariants` for both `TPDBClient` and `StashDBClient`) now live once in the `metadata` package, and both paths call them: the catalog live path (`stremio` `loadTpdbMeta`/`loadStashMeta`) and the MetaEnricher (`jobs` `runTPDB`/`runStash`). The two enrichment paths can no longer diverge, so UI covers match the catalog. Rate-limited TPDB still requeues; the catalog's cache read, miss-cache write, and Redis write-back are preserved.
- **PornRips catalog returned no streams on direct-backend Stremio installs** - the `pr_recent` (and `pr_studio`/`pr_tag`) catalogs emit `jstrm:` item IDs that carry only a PornRips detail URL (no infoHash), but `ServeStremioStream` only resolved `porndb:` and `hentai` IDs and returned an empty `streams:[]` for everything else. On an addon installed directly against the Go backend (bypassing the Node edge's debrid/P2P resolver), clicking a PornRips entry therefore showed no streams. `ServeStremioStream` now handles `jstrm:` (and `jstrg:` compact-studio group) IDs: a new `jstrmStreams`/`jstrmOrGroupStreams` path fetches the `.torrent` from the detail URL via `FetchTorrentData("pornrips", detailUrl, title)` and extracts the infoHash with `infoHashFromTorrent`, mirroring the existing `tpdbStreams` `porndb:` path; records that already carry an infoHash are emitted directly. The stream shape (`infoHash` + `behaviorHints.notWebReady`) matches `tpdbStreams`, so Stremio builds the magnet and plays via P2P. `jstrg:` groups decode each member to a `jstrm:` ID and resolve per-record, matching the Node edge `stream.ts` group path.
- **PornRips recent entries showed raw filenames despite having the real scene name in StashDB** - the `ReferenceWarmer` job seeded `shared_meta` under `source: "tpdb"` with WordPress reference data (the raw release filename as `title`, a pixhost `poster`, a "File Size/Duration" `description`), mislabeling WP data as a TPDB scene match. Because `MergeShared` is tpdb-first (`pickShared(tpdb.Title, stashdb.Title, "tpdb-first")`), the seed's raw-filename title overrode the real Stash scene name already stored in the `stashdb:pr:<slug>` doc, and because `enrichCatalogLiveMeta` skips entries that already have shared_meta (`noMerged=false`), on-demand TPDB resolution never replaced the seed. The result: entries that HAD resolved scene names in Stash still displayed the raw filename in `pr_recent`. `upsertReferenceSlug` no longer seeds the `tpdb` namespace when no real doc exists (the filename is already the `resolveCatalogReferenceMeta` fallback and the poster the `entriesToCatalog` `wp_poster` fallback, so the seed was redundant); real TPDB `shared_meta` is now written only by the live resolver and the `PornripsEnrich` sweep on an actual scene match. The dead write-only `SetReferenceMeta` Redis cache (no reader) and the `referenceToShared` helper were removed. A one-time cleanup deleted the 317 existing `tpdb:pr:` seed docs (no cast AND no tags - the seed signature; real TPDB matches always populate tags), so 249 entries with a Stash rescue now show their real scene name immediately (`JawBreakerz...Mckenzie.Mae...PRT` -> "Mckenzie Mae's Awesome BBC Experience") and the remainder live-resolve on view.
- **PornRips recent entries showed raw release filenames + WP featured images instead of TPDB/StashDB scene names + covers** - `pr_recent` listed releases like `MrLuckyPOV.22.11.04.Sophia.Locke...PRT` with the WP featured image because `buildMetas` reads the resolved name/cover from `shared_meta` (keyed `StableMetaID` = `"pr:<slug>"` for PornRips), and no shared_meta existed for PornRips entries. The on-demand live resolver `enrichCatalogLiveMeta` skipped any entry with a non-empty poster, and PornRips entries always carry a WP fallback poster, so they were never resolved live on view (the TPB parity gap - TPB entries have no poster initially, so they resolve). The gate now resolves PornRips entries live via the install's TPDB/Stash keys regardless of the WP poster (`shouldLiveResolve`), while other sources still skip once they have a real cover to avoid redundant TPDB calls. The `PornripsEnrich` sweep already called TPDB/Stash per entry but discarded the scene `Title` and wrote no shared_meta, so even enriched entries kept the raw filename; it now durably writes the resolved scene metadata to `shared_meta` (Redis + Mongo, keyed `"pr:<slug>"`) on each hit via `writePornripsSharedMeta`, mirroring the MetaEnricher `runTPDB`/`runStash` writes, so enriched entries show the scene name + cover instantly and the MetaEnricher dedup skips entries the sweep already filled. Result: enriched entries resolve from shared_meta (no live call); unenriched entries resolve on demand on view and get durabilized by the sweep/MetaEnricher; TPDB-unmatched entries still fall back to the raw filename + WP poster (no empty catalog).
- **PornRips Studio/Tag catalogs returned no/few entries** - the durable `pornrips_entries` store only assigned `studio` from a TPDB scene match's `site.name`, and TPDB doesn't cover most PornRips releases, so most entries kept `studio=""` and `pr_studio`/`pr_tag` filtered to near-nothing. The studio is in fact the post's WordPress `post_tag` (PornRips tags each post with its site, e.g. "BrazzersExxtra"; WP `categories` are only quality buckets like 720p/1080p). Ingest now embeds `wp:term` and stores the `post_tag` name as `studio`/`studio_norm` (`$set`, so a re-walk backfills entries ingested before this), and enrichment no longer overwrites it - the WP tag is the authoritative studio source and `NormToken`-matches the curated `pr_studio` list ("BrazzersExxtra" == "Brazzers Exxtra"). `pr_tag` had no WP source (WP has no content-category taxonomy, only studio tags), so its options are now built dynamically from the TPDB/Stash tag vocabulary actually present in the store (top-N by count via `GetPornripsTopTags`), falling back to the curated taxonomy when the store is cold - every option then maps to real enriched entries instead of the fixed list that returned empty for many categories.
- **PornRips Studio/Tag filters returned the Recent feed** - the `pr_studio`/`pr_tag` Discover catalogs carry the selection as the `genre` extra, and the filter chain tries the WordPress REST API first (`FetchPornripsCatalog`). That call only added a `categories`/`tags` filter when the option name resolved to a WP term ID; when `resolveTermID` returned 0 (common, since many studio/tag names don't match a WP term name or slug - e.g. "5K Porn", "BANG!", "ATKingdom", "MILF AF") it fetched **unfiltered** recent posts, and the caller treated that non-empty result as the final filtered answer and never fell through to the token-scan or WP `?s=` paths that actually filter by the name. `FetchPornripsCatalog` now returns nothing when the term doesn't resolve, so the chain reaches the WP `?s=<studio/tag>` scrape search that filters by name. `resolveTermID` also caches the miss so paginating an unresolved filter doesn't re-query the term search endpoint on every page.
- **No-date extra-indexer covers (alias soups without prefix/date)** - extra-indexer results (bitsearch/knaben_adult/xxxclub) drop the "OnlyFans" prefix and the date, leaving a flat performer-token soup like "June.Liu.SpicyGum.Juneliu.Emiri.Momota.Mizukawasumire.BGG.threesome.mp4" or the "aka"-joined "OnlyFans.Emiri.Momota.aka.Mizukawa.Sumire.June.Liu.aka.JuneLiu.SpicyGum.BG...MeetMyLove.mp4". `SearchMetadataProbe` now runs the no-date pair probes FIRST (before the parse/q loop that can't match a token soup), gated by `VerifyPairDescriptor` (both probe performers match distinct candidate performers + a scene descriptor token overlaps the candidate title, the disambiguator the missing date would otherwise provide). `performerPairProbes` is generalized (sliding 2-token name windows, capped at 6) and re-prepends the studio token for no-date titles.
- **Pair-probe lookup timeout on slow TPDB round-trips** - the no-date pair-probe sweep fires up to 6 `q=` keyword calls, and from the deploy region each round-trip is ~1-2s, so a sequential sweep rode the 8s `loadTpdbMeta`/`loadStashMeta` deadline too closely. The probes now fan out concurrently (staggered by the existing minGap throttle, so not a burst), resolving in one round-trip with margin to spare; the MetaEnricher also drains faster, so non-priority no-date items are no longer starved behind the backlog.
- **Live meta handler no-date soup timeout + TPDB 429 storms** - `loadTpdbMeta` looped over every `metadataTitlesForLookup` query variant and called `SearchMetadataProbe` per variant, but for a no-date performer soup the pair-probe branch keys on the fixed parsed release (not the query string) and returns early, so each variant re-fired the identical 6-probe fan-out: 6xN TPDB `q=` calls per lookup, which burned the 8s deadline (the live handler returned "no cover" where a single call would have resolved) and tripped TPDB 429s that also slowed the background warmer sharing the same API key. `loadTpdbMeta` now collapses to a single `SearchMetadataProbe` call for no-date pair-probe titles (`NoDatePairProbeTitle`: `Date=="" && len(performerPairProbes)>0`, the exact condition `SearchMetadataProbe` itself uses to take the early-return branch), so the live path resolves in one round-trip like the warmer and stops multiplying TPDB calls. Dated titles, dash-form OnlyFans, and normal studio+date releases keep the variant loop unchanged.
- **Miss-cache poisoning on transient TPDB/StashDB failures** - `loadTpdbMeta`/`loadStashMeta` cached a 1h "miss" sentinel on ANY no-match, including a lookup timeout (the 8s deadline makes `SearchMetadataProbe` return `nil,nil`, not an error) or a TPDB 429. A transient timeout or rate limit then blocked the cover for an hour even though the next attempt would resolve it. Misses are now cached only on a confirmed clean no-match: `!rateLimited && rctx.Err() == nil` (TPDB) / `rctx.Err() == nil` (StashDB, which has no rate-limit error type). Timeouts and 429s leave the key absent so the warmer and the next live hit re-probe.
- **Auth: stop logging users out on refresh during transient store errors** - `/api/auth/validate` and `/api/auth/exchange` returned `401 INVALID_TOKEN` on ANY session-store error, including the brief window after a backend redeploy when the Mongo pool is not yet ready. The frontend then wiped the still-valid 30-day `sessionToken` cookie and forced a full Google re-login on the next refresh. A transient store error now returns `503 STORE_UNAVAILABLE` (so the client retries / keeps the token); only a genuinely nil session returns `401`.
- **Google OAuth: stop forcing the consent screen on every login** - `prompt=consent` forced the full Google consent screen (and the "unverified app / trust this site" warning while the consent screen is in Testing status) on every re-login, which recurred whenever a refresh logged the user out. Changed to `prompt=select_account` so returning users pick/switch their account without re-authorizing. Publish the OAuth consent screen to "In production" in the Google Cloud Console to remove the unverified-app warning for first-time logins (non-sensitive scopes, no verification needed).

- **Favorites covers for old favorites (root cause)** - `GetFavorites` only resolved TPDB/StashDB covers from `shared_meta` for favorites that had an existing `images` row, iterating `coverImages`. Old favorites (older than the durable-cover-persist fix) have no `images` row - their only cover is an expired `temp/` presigned URL stored in `torrent_data` - so `coverImages` was empty, the recovery loop never ran, and the UI fell back to the stale dead URL. `GetFavorites` now also resolves `shared_meta` by metaID (lowercase infoHash from the magnet) for favorites with no `images` row, surfaces the recovered poster as `coverImage`, and promotes it to a durable `keep/` row in the background so future loads find it. This is why the cover-image endpoint returned the TPDB poster while the favorites endpoint showed a blank/temp cover for the same torrent.
- **MetaEnricher priority upgrade on re-enqueue** - `EnqueueMetaLookups` dedup path updated the title/detailURL/website of an existing queued item but not its `priority` flag, so a torrent first enqueued by browse (non-priority) and later re-enqueued by a favorites load (priority) stayed non-priority and drained behind a multi-thousand-item backlog instead of ahead of it. Favorites covers for TPDB-matchable scenes never surfaced. The dedup now upgrades `priority` when a re-enqueue marks the item priority.
- **Stale empty-poster shared_meta entries (root cause of remaining temp covers)** - `ExistsManyShared` checked only Redis key existence, so a `tpdb-shared:`/`stashdb-shared:` entry written with an empty poster by a pre-clean-query wrong/no-poster match counted as "already enriched". The MetaEnricher then short-circuited (`hasTPDB && hasStash`) and never re-probed, while `cover-image` read the empty poster and returned "not found" - leaving TPDB-matchable mainstream favorites (Blacked, JulesJordan, RickysRoom) stuck on expired temp covers even though TPDB resolves their scenes. `ExistsManyShared` is now poster-aware (MGET + check `Poster != ""`), so empty-poster entries are treated as absent and re-probed with the now-correct clean query, overwriting them with the real poster.
- **Redis->Mongo shared_meta rehydrate (cover-image reads Mongo)** - the MetaEnricher only rehydrated Redis from Mongo, never the reverse. The stremio live meta handler caches TPDB/StashDB posters to Redis only, and a partial write that succeeded in Redis but failed in Mongo left favorites with a poster in Redis but no durable Mongo row. Since `hasTPDB`/`hasStash` stay true from the Redis entry, the item was never re-probed and `cover-image` (which reads the Mongo store) returned "not found" despite Redis holding the poster. The MetaEnricher now also copies Redis->Mongo when Redis has the poster and Mongo does not, so the store `cover-image` reads is populated and the cover resolves.
- **TPDB query fix (cover recovery root cause)** - `SearchMetadataProbe` now sends the parsed clean query (studio + cleaned scene, date/resolution stripped) instead of the raw torrent title, and progressively drops trailing words when TPDB returns nothing. TPDB's parse endpoint returned zero results for messy titles like "Blacked 26 06 12 Hope Heaven Pull Chapter 5 First 2160p MP4" but resolves the same scene from "Blacked Hope Heaven Pull Chapter 5"; VerifyMatch then accepts it via studio + exact date + performer overlap. This was the cause of `tpdbHits: 0` in MetaEnricher runs - mainstream scenes (Blacked, JulesJordan) that ARE in TPDB were never matched, so shared_meta never populated and favorites with deleted temp S3 covers had no TPDB fallback.
- **TPDB covers for flat multi-performer OnlyFans titles** - flat OnlyFans releases whose scene field is a string of co-performer aliases (e.g. "OnlyFans 23 06 17 June Liu SpicyGum Juneliu Emiri Momota Mizukaw", where SpicyGum and Juneliu are both June Liu and Mizukaw is Sumire Mizukawa) never resolved a TPDB cover: the parse endpoint returns nothing for the alias soup and the strict-AND `q=` keyword endpoint rejects it too, so the canonical performer pair the scene is indexed under ("June Liu Emiri Momota") was never formed. `SearchMetadataProbe` now falls back, after the parse loop misses, to `q=` probes that pair the first 2-token performer with each later 2-token group (capped at 4); one probe lands on the canonical pair and `VerifyMatch` rejects the wrong-group hits. Covers for these titles now populate via both the MetaEnricher (30-day durable cache) and the live on-demand meta handler (fresh searches).
- **MetaEnricher priority queue** - favorites enqueues are marked priority and drained ahead of the browse/search backlog. The shared queue grows faster than it drains (enqueue outpaces the 250/tick drain), so without priority, user-facing favorites covers enqueued last would never surface. `MetaEnqueueItem` gains a `priority` field; `drain` partitions priority-first. Also bumped queue cap 5000->20000 and maxPerTick 150->250 so favorites enqueues are no longer dropped at the cap.
- **enrichCoverImages TPDB enqueue** - browse/search endpoints now enqueue async TPDB/StashDB lookup for results with no shared_meta cover; covers appear on the next page load (seconds) instead of waiting up to 8 hours for the DescriptionImageCache job.
- **GetFavorites cover recovery** - re-signs expired S3 URLs, resolves TPDB/StashDB covers from shared_meta using the correct metaID (infoHash from magnet), promotes found covers to keep/ async, and enqueues MetaEnqueuer TPDB lookups for favorites whose temp S3 objects have been deleted so covers appear on next load.
