# Changelog

All notable changes to **torrent-search-go** (Stremio protocol API, scrapers, jobs) are documented here.

Format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).
Stremio manifest versions track the addon release line (`internal/stremio/manifest.go` `addonVersion`, overridable by `ADDON_VERSION`). The source of truth for the shipped addon version is `tpb-stremio-addon/package.json`.

Cloudflare-protected stream hosts are handled Stremio-natively via `behaviorHints.proxyHeaders.request` (`User-Agent` + `Referer`), no in-addon proxy. Data model: `pkg/models.PerverzijaEntry` + `FreepornvideosEntry` (flat stores `perverzija_entries` / `freepornvideos_entries`, not grouped like pornrips); Stremio ids `pvz:{slug}` / `fpv:{numericID}` added to manifest `idPrefixes`. The Mongo-CPU pitfall (distinct aggregation on the manifest path drove mongod to 100% in the pornrips era) is avoided: top-N studio/tag/performer option lists are precomputed by each source's sync job into Redis KV blobs (`cache.PrefixPerverzijaGenres` / `PrefixFreepornvideosGenres`, 7-day TTL), read as two cheap Redis GETs at manifest time (`loadTubeGenreOptions`); cold installs (no Redis / no precompute) hide the Studio/Tag/Performer catalogs and show only Recent + Search. Scrapers held as typed `Runner` fields (not registered in the torrent `Scraper` service - they would pollute `SearchAll`) and exposed to the stremio `Handler` as concrete `*scraper.PerverzijaScraper` / `*FreepornvideosScraper` to avoid the stremio/jobs import cycle. Local-only bulk-fill CLIs `cmd/pvingest` / `cmd/fpvingest` (gitignored, mirror `cmd/pringest`) walk the full archive then enrich to completion. Catalog names use the `·` separator (no em/en dashes). Stream URLs extracted from upstream HTML/m3u8 are validated via `ResolveSafeStreamURL` (scheme `http`/`https` only, host not private/loopback/link-local/CGNAT/metadata, `localhost` hostname blocked) before they are emitted to Stremio - a compromised upstream cannot inject an internal URL (`169.254.169.254`, `localhost`, `192.168.x.x`, ...) that the end-user's Stremio streaming server would fetch on their behalf via `proxyHeaders.request` (the backend never dials these URLs itself, so the existing SafeClient dial-time SSRF defense does not cover this path). The xtremestream hash regex is length-capped (`{20,200}`) so a compromised detail page cannot store a megabyte-scale hash. Regression tests: `TestParseHLSVariantsRejectsInternalURL`, `TestFreepornvideosResolveStreamRejectsInternalURL`. Ingest is 1-indexed (`IngestPage(ctx, cursor+1)` mirroring hentai's `ListSeries(page+1)`) so a fresh KV cursor (0) fetches page 1, not page 0 (WP REST returns 400 for page 0, which would otherwise be treated as end-of-feed and the collection would never populate). The "All" tag genre falls back to Recent (`filterNonEmpty` drops the empty token the catalog sends for "All" before the `$in`, so `$in: [""]` - which matches zero docs - is never queried; studio/performer already fell back at the store level). Permanently-gone detail pages (HTTP 410/404) are marked `detail_scraped` via an `errPageGone` sentinel so deleted posts do not accumulate at the head of the newest-first enrich sweep and livelock it; transient failures (5xx/Cloudflare/timeout) still return a plain error and retry next tick (the pornrips enrich philosophy). The same `errPageGone` on the FreePornVideos listing walk (`IngestPage`) is treated as end-of-feed (empty slice, nil error) - pages past the archive end return 404, not 200-empty, so without this the ingest loop would retry the gone page forever and never reach `hitEmpty` (the livelock that bit the real backfill at page 6001); regression test `TestFreepornvideosIngestPageGone`. Meta cast/studio search links use `url.QueryEscape` so a name with `&`/`#`/`?` does not split the `stremio:///search?search=` query.

### Changed
- **Tube-source ingest sweep batched via BulkWrite (WAN-batching fix)** - the watchporn and yesporn ingest sweeps (`WatchpornIngest` / `YespornIngest`) wrote one `UpdateOne` per card, which on the Mac one-time bulk fill is WAN-bound (Mac -> prod Mongo ~280ms/round-trip -> ~210 entries/min -> ~9h projected for the watchporn ~139k archive). They now buffer each page's cards into one `BulkWrite` (500/batch, unordered, mirroring `BackfillPornripsSceneGroup`) via new `UpsertWatchpornEntries` / `UpsertYespornEntries` store methods that reuse the per-doc `$set`/`$setOnInsert` builder (`watchpornUpsertUpdate` / `yespornUpsertUpdate`) so the enrichment-protection contract is byte-identical (`date`/`duration`/`poster`/`excerpt`/studios/tags/performers(+`_norm`)/`description`/`detail_scraped` stay `$setOnInsert`; only `video_id`/`slug`/`title`/`detail_url`/`has_4k`/`website`/`updated_at` are `$set`). Cut the watchporn Mac -> prod-WAN fill to ~1100/min (~5x; the 139k archive ingested in ~1.7h, enriched in ~3h at concurrency 10). The prod sync tick (LAN) uses the same batched call as a no-op-cost superset (one round-trip for ~35 entries). Partial-failure tolerant: the Go driver returns counts alongside a `mongo.BulkWriteException`, so per-doc write errors are logged + skipped (`SetOrdered(false)` continues); a connection error drops the 500-doc batch + logs (strictly better than 500 failing per-entry round-trips). Empty/empty-VideoID input is a no-op. `UpsertWatchpornEntries` returns total docs touched (`UpsertedCount + MatchedCount`); `UpsertYespornEntries` returns NEW inserts only (`UpsertedCount`) because the yesporn ingest loop needs 0-new as its end-of-feed signal (see the Fixed entry). Also added `cmd/yesporningest` (gitignored, mirrors `cmd/watchporningest`) for the yesporn Mac one-time fallback/verification: `-bulk` loops ingest (terminates via the dedup end signal) + enrich, default mode runs one recent tick; no Redis wiring (prod's `YespornSync` owns the genre blob).
- **Tube sources refactored to a shared `TubeSource` interface + registry (Phase 0 of the tube-source expansion)** - Perverzija and FreePornVideos were fully copy-pasted per source (~15 near-duplicate files each, no shared contract). They now share one generic catalog/meta/stream handler stack. New `internal/stremio/tube_source.go` defines the `TubeSource` interface (`Key`/`IDPrefix`/`CatalogPrefix`/`DisplayName`/`CachePrefixes` + read methods + `ResolveStream`/`StreamReferer` + Phase-2 `HasSiteSearch`/`SiteSearch`) and a `TubeSourceRegistry` keyed by source key, id prefix, and catalog prefix. `internal/stremio/tube_catalog.go` generalizes `servePerverzija*`/`serveFreepornvideos*` into `serveTubeCatalog`/`serveTubeMeta`/`serveTubeStream` and is the new home for the shared `releaseYear`/`tubeMetaLinks`/`tubeStreamsToStremio` helpers. Each source is now one adapter file: `tube_source_perverzija.go` and `tube_source_freepornvideos.go` (the `*Store`/`*StreamResolver` interfaces moved here too) map their per-source entry to the read-side `TubeEntry` superset (`pkg/models/types.go`) at read time - no BSON backfill. Dispatch in `catalog.go`/`meta.go`/`tpdb_catalog.go` is registry-driven (`h.TubeSources.LookupByCatalogPrefix` / `LookupByIDPrefix`) for `pvz_`/`fpv_` catalogs and `pvz:`/`fpv:` ids (and any future tube source). The manifest is registry-driven: `tubeCatalogDefs(src, opts)` replaces `perverzijaCatalogDefs`/`freepornvideosCatalogDefs`, the catalog-registration block loops `tubeSources`, `idPrefixes` is built from each source's `IDPrefix()`, and `tubeGenreOptions` is now `map[string]tubeGenreSet` keyed by source. `normalizeSources` and `isCatalogAllowed` read a package-level `defaultTubeRegistry` (set once at `main.go` wiring via `SetDefaultTubeSourceRegistry`) so adding a source = register it, with no edit to those functions. This unblocks adding 6 more tube sources + a cross-source `tube_search` as one-adapter-plus-Register work (Phase 1-3).

### Removed
- **Dead `HasSiteSearch`/`SiteSearch` tube-source scaffolding deleted (Phase 2.1)** - the `TubeSource` interface (`tube_source.go`) carried two methods, `HasSiteSearch() bool` and `SiteSearch(ctx, query, skip, limit)`, intended for a live per-source search endpoint. All five adapters (perverzija, freepornvideos, yesporn, watchporn, porneec) stubbed them `return false` / `return nil, nil`, and `serveTubeCatalog`'s `_search` branch gated `if src.HasSiteSearch() { src.SiteSearch(...) } else { src.Search(...) }` - always taking the `else` (Mongo-regex `Search`) path. The methods were dead code: tube search already works via Stremio's native fan-out across the per-source `_search` catalogs (each is `search: true` in the manifest, and Stremio's search page queries every `search:true` catalog and merges client-side), so no source ever needed a live site-search endpoint and a separate cross-source `tube_search` catalog would have duplicated that native fan-out. Both methods are removed from the interface + all five adapters, and the `_search` branch is simplified to always `src.Search(ctx, q, skip, limit)` (the empty-query-returns-empty guard is preserved). No behavior change (the dead branch was never taken); the existing `tube_catalog_test.go` `pvz_search` empty-query regression test still passes. `go build ./...` + `go vet ./...` + `go test ./...` clean. Interface + package doc comments updated to describe the Mongo `Search` backing + the native-fan-out rationale.
- **Dead `setTubeGenreOptions` Redis helper deleted** - the tube-source genre-option blobs are written directly by each source's sync job into `PrefixPerverzijaGenres` / `PrefixFreepornvideosGenres`, so the unused `setTubeGenreOptions` method on the Redis store was dead code; deleted.
- **Bitsearch indexer (`bitsearch.eu`) dropped** - the `BitsearchScraper` (`internal/services/scraper/bitsearch.go`) and `BitsearchProvider` (`internal/services/streams/providers/bitsearch.go`) are deleted along with their `main.go` registrations. `bitsearch` is removed from the `ExtraIndexers` fanout (`catalog.go` `fanoutAdultSources` + `catalogFanoutKey` `"bs"` tag + `filterByIndexerConfig` case), `tpdb_catalog.go` `matchedSourcesFilter`, the `ENRICHED_SCENES_SOURCES` default (`enriched_scenes_sync.go`), and the magnetio defaults (`config.go` `defaultConfig.Providers` + the `"s13"` public provider alias). Saved magnetio URLs still carrying `s13` resolve to no-op (the alias is gone, the unknown id is skipped downstream). Knaben/xxxclub/mypornclub and the rest of the extra-indexer set are unchanged.

Two fixes: (1) `YespornIngest` now treats "0 NEW inserts in a tick" (every card already in the store, via `UpsertYespornEntries` returning `UpsertedCount` only) as the archive-fully-covered end-of-feed signal - resets the cursor to 0 and returns `hitEmpty=true`; a transient all-matched tick before full coverage is statistically unreachable (a 2000-card random subset of the ~989 archive is all-already-seen only at ~100% coverage), so a single 0-new tick is reliable. Only `tpdb_new` is busted (query-keyed `tpdb_search` pages re-probe live TPDB on a miss and are cheaply re-cached on demand) and only when `resolved > 0` so a clean-miss tick does not churn the cache. `DelByPrefix` is nil-safe. The key-format change also naturally orphans the old keys (which held the future-dated scenes the date-filter fix now drops), so they TTL out without a manual flush.
- **tpdb_new / tpdb_cat / stashdb_cat catalogs dominated by future-dated (pre-release) scenes** - TPDB stores scenes with a future release date that sat at the top of the catalogs' `date:-1` sort, so recently-released scenes never surfaced and the catalogs looked like they were not updating. `GetEnrichedScenesByMatchedSources` (`pkg/mongo/enriched_scenes.go`, the store-backed browse + category catalog query) now adds `date $lt tomorrow` (tomorrow in container UTC) so pre-release scenes are dropped and the newest already-released scenes show first. `$lt: tomorrow` (not `$lte: today`) keeps same-day scenes; `date` is TPDB's raw `YYYY-MM-DD` string so the lexicographic compare matches calendar order, served by the existing `{source:1, matched_sources:1, date:-1}` index. Deliberately NOT applied to `GetEnrichedScenesByMatchedSourcesAndIDs` (tpdb_search), since a search returns all matched scenes for the query regardless of release date.

Builders extracted (`perverzijaUpsertUpdate` / `freepornvideosUpsertUpdate`) so the field-placement contract is unit-testable without a live Mongo. ### Security
- **Auth bypass via unsigned "temporary token" removed (Critical)** - `internal/middleware/auth.go` accepted any base64-encoded `{id,email,name,timestamp}` JSON blob as a valid session with no signature check, letting an attacker forge any user identity and take over their account (favorites, Real-Debrid key, all `RequireAuth` routes). The `validateTemporaryToken` fallback and its base64/email forging path have been deleted; only database-issued sessions validate now.
- **Monitoring endpoints fail-closed in production (Critical)** - `/api/monitoring/*` (IP block/unblock, cache bust, log viewer, job triggers, debug-favorites) shipped with zero auth when `DASHBOARD_PASSWORD` and `MONITORING_IP_ALLOWLIST` were unset, the documented default. `RequireDashboardAuth` (`internal/middleware/dashboard_auth.go`) now denies all monitoring traffic in production when no password is configured; dev/local stays open.
- **SSRF defense on all outbound scraper traffic (Critical)** - a user-supplied `torrentUrl` path segment reached scrapers (`nyaa`, `hiddenbay`, `pornrips`, ...) as a raw request URL, enabling server-side requests to `169.254.169.254`, internal services, and localhost. New `internal/services/scraper/safehttp.go` provides `NewSafeClient`: a dial-time IP filter (refuses private/loopback/link-local/CGNAT/metadata ranges, pins the resolved IP to defeat DNS rebinding), a `CheckRedirect` that re-validates each hop, and an 8 MiB response-body cap. Every scraper constructor and the shared `Service` client now route through it.
- **SSRF via unresolved hostname in emitted stream URLs (High, found in review)** - `ResolveSafeStreamURL` (`internal/services/scraper/safehttp.go`) validated the parsed URL host and rejected literal private/loopback/link-local/CGNAT/metadata IPs, but a hostname that DNS-resolves to a private IP (e.g. `internal.example.com` -> `169.254.169.254`) passed through, because the backend emits the stream URL to Stremio (the end-user's streaming server dials it via `proxyHeaders.request`) and never dials it itself, so the existing `safeDialContext` dial-time SSRF guard does not cover this path. A compromised upstream detail page could inject such a hostname and pivot the end-user's Stremio server into the metadata service. `ResolveSafeStreamURL` now resolves the hostname via DNS (5s timeout) and rejects the URL if any resolved IP is private/loopback/link-local/CGNAT/metadata; unresolvable hostnames fail closed. A package-level `resolveStreamHost` var is the test seam. Test: `TestResolveSafeStreamURLRejectsPrivateDNS`.
- **CRLF header injection via freepornvideos slug (High, found in review)** - the freepornvideos card-slug regex `([^/]+)` accepted `\r\n` (goquery decodes `&#13;&#10;`), so a compromised upstream detail URL could embed CRLF in the slug, which became the `Referer` value emitted in `behaviorHints.proxyHeaders.request`, letting an attacker inject `X-Injected: evil` headers into the end-user's Stremio stream request. The card parser now drops any card whose slug contains a control char at ingestion (`StripHeaderUnsafe`), and `tubeStreamsToStremio` strips CR/LF from the referer at emission as defense-in-depth (the referer is the entry's stored detail URL, which could carry CRLF injected before the ingestion-time guard existed). Test: `TestFreepornvideosCardDropsCRLFFromSlug`.
- **Stremio `:config` segment redacted from logs and the API tracker (High)** - `/stremio/:config/...` and `/magnetio/:config/...` URLs carry the user's Real-Debrid / TPDB / StashDB / media-flow keys in the path; every catalog/meta hit wrote them into `app.log`, `all.log`, and the `api_usage` Mongo collection. `redactPath` (`internal/middleware/router.go`) replaces the config segment with `:redacted` before logging or tracking.
- **`X-Forwarded-For` / `X-Forwarded-Proto` gated on `TrustProxy` (High)** - `clientIP`, `isHTTPS`, and the IP-allowlist resolver honored proxy headers unconditionally, so a direct client could spoof its IP past the rate limiter, DDoS guard, and monitoring allowlist, or force HSTS over plain HTTP. A package-level `trustProxyHeaders` flag (set from `cfg.Security.TrustProxy` via `WithTrustProxy`, now actually wired instead of the prior no-op stub) gates all three.
- **CORS no longer reflects arbitrary origins with credentials (High)** - when `FRONTEND_URL`/`ADDITIONAL_CORS_ORIGINS` were unset, config fell back to `["*"]` and the middleware reflected the requesting origin alongside `Access-Control-Allow-Credentials: true`, letting any website issue authenticated cross-origin requests with the victim's session cookie. `isOriginAllowed` (`internal/middleware/cors.go`) now ignores `*` when credentials are enabled and requires an explicit origin match.
- **Global request-body cap (High)** - ~22 JSON endpoints had no `MaxBytesReader`, so a single multi-GB body could exhaust memory. A new `MaxBodySize` middleware + `WithMaxBodySize` router option caps every request at 2 MiB (`main.go`); `KVSet` additionally bounds a single cache value at 256 KiB so one user can't bloat the shared KV collection.
- **IDOR on favorite entry mutations closed (Medium)** - `UpdateFavoriteEntryCoverImage` and `UpdateFavoriteEntryMagnetLink` filtered Mongo by `_id` only, so any authenticated user could overwrite another user's cover URL or swap their magnet to a malicious torrent by guessing the entry id. Both storage methods now take a `userID` and filter `{_id, user_id}`; the handlers pass the authenticated user's id.
- **Constant-time `ADDON_API_TOKEN` comparison (Medium)** - `MatchesAddonToken` compared the service token with `==`, exposing a timing oracle. Switched to `subtle.ConstantTimeCompare`.

- **Miss-cache poisoning on transient TPDB/StashDB failures** - `loadTpdbMeta`/`loadStashMeta` cached a 1h "miss" sentinel on ANY no-match, including a lookup timeout (the 8s deadline makes `SearchMetadataProbe` return `nil,nil`, not an error) or a TPDB 429. A transient timeout or rate limit then blocked the cover for an hour even though the next attempt would resolve it. Misses are now cached only on a confirmed clean no-match: `!rateLimited && rctx.Err() == nil` (TPDB) / `rctx.Err() == nil` (StashDB, which has no rate-limit error type).
